BlockHawk provides specialized detection for Solana-specific attack vectors. This document details the types of attacks detected, why they're considered anomalous, and guidelines for handling each threat type.

1. Price Oracle Manipulation Attacks

HIGH SEVERITY
Detection Method:
  • Multiple transactions affecting the same price oracle in rapid succession
  • Unusual trading volumes against specific liquidity pools
  • Z-score analysis of price movements exceeding historical volatility by 3-5Ïƒ
Why It's An Anomaly:

When BlockHawk detects a series of trades causing price movements that statistically deviate from normal patterns (especially if from new or rarely-used addresses), it signals potential market manipulation.

Handling Guidelines:
  • Immediately pause dependent protocols' liquidation functions
  • Alert protocol admins via priority channels
  • Implement circuit breaker to temporarily switch to TWAP oracles
  • Analyze transaction sequences to identify manipulator addresses
Example Protection:

During the Mango Markets incident, this pattern would have triggered alerts within the first few transactions of the $114M exploit, potentially providing time to pause withdrawals.

2. Account Data Validation Attacks

HIGH SEVERITY
Detection Method:
  • Unusual account ownership relationships
  • Non-standard account data size changes
  • Unexpected account type transitions
Why It's An Anomaly:

Legitimate transactions rarely involve unexpected changes to fundamental account properties. Pattern recognition identifies when accounts are used in ways that deviate from their designed purposes.

Handling Guidelines:
  • Flag and block transactions with suspicious account relationship patterns
  • Alert developers to potential validation bypass attempts
  • Temporarily restrict functions involving affected accounts
  • Implement emergency fixes to validation logic
Example Protection:

BlockHawk would identify the Cashio infinite mint vulnerability by detecting abnormal account relationship patterns before significant funds were extracted.

3. Cross-Program Invocation (CPI) Attacks

MEDIUM SEVERITY
Detection Method:
  • Unusual CPI call sequences
  • Unexpected program interactions
  • Deviation from normal program invocation patterns
Why It's An Anomaly:

Programs typically interact with a predictable set of other programs in specific sequences. Unusual call patterns often indicate attempts to exploit trust relationships.

Handling Guidelines:
  • Implement program allowlisting for sensitive operations
  • Block unusual CPI sequences at protocol level
  • Require multi-signature approval for sensitive program interactions
  • Review and restrict program authority relationships
Example Protection:

By analyzing program invocation patterns, BlockHawk can identify when attackers attempt to exploit cross-program permission inheritance issues.

4. Transaction Frequency Anomalies

MEDIUM SEVERITY
Detection Method:
  • Statistical analysis of transaction frequency per address
  • Time-series analysis of transaction patterns
  • Adaptation to normal behaviors for different address types
Why It's An Anomaly:

Addresses typically exhibit consistent transaction patterns. Sudden bursts of activity from previously quiet addresses or unusual timing patterns often precede attacks.

Handling Guidelines:
  • Implement progressive rate limiting for suspicious addresses
  • Require additional verification for high-frequency transactions
  • Apply time-delay for large withdrawals following unusual activity
  • Monitor transaction destination patterns
Example Protection:

BlockHawk detected a transaction frequency anomaly during beta testing where an attacker attempted a series of rapid withdrawals from a compromised wallet.

5. Compute Unit Exhaustion Attacks

MEDIUM SEVERITY
Detection Method:
  • Monitoring compute unit consumption patterns
  • Detecting transactions that approach but don't exceed limits
  • Identifying unusual instruction combinations designed to consume resources
Why It's An Anomaly:

Normal transactions rarely optimize for maximum compute unit consumption. Patterns that consistently consume near-maximum resources often indicate attempts to bypass security checks.

Handling Guidelines:
  • Reserve additional compute units for security checks
  • Implement tiered security measures with varying compute requirements
  • Flag transactions with suspicious resource consumption patterns
  • Consider redesigning critical security checks to use fewer compute units
Example Protection:

By detecting compute unit patterns common in bypass attempts, BlockHawk helps protect against attacks that attempt to exhaust resources before security checks run.

6. PDA (Program Derived Address) Manipulation

HIGH SEVERITY
Detection Method:
  • Tracking PDA derivation patterns
  • Monitoring seed manipulation attempts
  • Detecting unexpected address relationships
Why It's An Anomaly:

PDAs are derived deterministically and should follow predictable patterns based on program logic. Attempts to manipulate or forge unexpected PDAs often indicate exploit attempts.

Handling Guidelines:
  • Validate all PDA relationships thoroughly
  • Implement additional checks for sensitive PDAs
  • Monitor for attempts to create PDAs with unusual seeds
  • Review cross-program PDA access patterns
Example Protection:

The system can detect anomalous PDA derivation attempts that might indicate someone trying to gain unauthorized access to program-controlled accounts.
